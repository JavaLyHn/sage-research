> 这一版整合了**公开页体验 + 登录后实测 + 6 个 agent 详情页 skill 清单 + 真实用户对话历史**四层证据。
> 体验对象：https://www.octok.com/ ｜ 输出日期：2026-05-25
>
> 与 v1（`opinion.md`，仅公开页）对比：登录后揭示了 7 个**全新的产品级问题**。
>
> ⚠️ **v3 修正说明**：v2 的发现 3 和发现 7 因证据不足下了错误判断——
> 实际上 6 个 agent **都有独立详情页（`/team/<agent>`）+ Skills tab + 快速指令清单**，
> 总共 **23+ 个 skill**，产品工具体系比 v2 想象的完整得多。v3 已重写这两条发现，
> 并把**真正的产品意见**修正为「skill 是真实存在的，但**执行过程对用户黑盒**」。

## 总判断：升级版

> **Octok 的产品骨架比公开页给人的印象完整得多**——工作台真实存在，6 位 AI 员工是真的可对话，
> 用户对话历史是真的，**每位员工都有独立详情页 + Skills tab + 5-6 个具名快速指令**（Alice 5 个 / Sophia 3 个 / William 4 个 / Daniel 5 个 / Ethan 6 个），OKR 与任务卡都有落地。
>
> **真正的问题不是没有 skill，是 skill 的执行过程对用户完全黑盒**：
> 用户点击「生成财务诊断」后看不到调了什么工具、读了哪些数据、产出存在哪里；「动态」流全部「暂无动态」；
> 知识库结构混乱（4 套存储概念并存）；6 位员工里 5 位被付费门槛锁着却不告诉你哪几个；
> 设置 / 计费 / 订阅模块根本没建。
>
> **Octok 不是「营销大于产品」，是「能力做了 80% 但展示了 40%」**——降低承诺 + 把已有能力的执行透明度做出来，
> 就能让现状显得更好。

## 登录后才能看到的核心发现（v1 看不到的）

### 🆕 发现 1：「控制台」其实只是「员工选择器」，不是工作台

**实测**：
登录后落地 `https://www.octok.com/` 上的工作台首屏文案是「**选择一位 AI 员工开始对话**」。整个首屏是 6 张员工卡片的网格 + 3 条预置问题（"解读 Navigator 诊断要点 / 制定季度出海 OKR / 分析韩国市场机会"），**完全没有**：
- 最近对话 / 进行中任务 / 本周产出
- 关键指标仪表盘 / 进度条
- 任何「团队工作进展」的可视化

但顶部导航写着「控制台 / 工作日历 / AI 员工 / 企业资料 / 能力 / 新增对话」——这个导航暗示这里应该是「指挥中心」，**实际上只是一个对话入口选择器**。

**真实判断**：
官网首页演示截图里的「**指挥中心 + 工作日历 + 5/6 员工运行中 + 12 个任务进行中**」是**精心制作的演示画面**，真实产品里这一层根本不存在。这不是漏洞或缺陷，是产品没做。

**解决方案**：
两条路二选一，**不要两条都做**：
- 🅰️ 把「控制台」重命名为「我的团队」，承认它就是对话选择器，从首页删掉「工作日历」和「运行中员工数」的演示截图
- 🅱️ 真做一个工作台把演示画面兑现：最近 3 个对话 + 本周产出文档列表 + 进行中任务进度条 + 6 位员工在线状态

🅱️ 是正解但工作量大（2-3 个月）；🅰️ 是 1 周内可以做的诚实降级。**先做 🅰️，3 个月后再做 🅱️**。

### 🆕 发现 2：「部分员工需先雇佣后解锁对话，便于演示订阅与权限流程」

**实测**：
工作台首屏底部有这么一行小字。这一句直接承认了「免费+付费分层」的门禁存在——但是：
- **没说哪几位是免费的**（推测 Navigator 是免费入口，但不确定）
- **没说哪几位收费**（点了才知道撞墙）
- **没说雇佣价格**（工作台内也没有定价入口，见发现 6）

**真实判断**：
这是个**双重失败**：
1. 承认了商业模式存在（不再「全免费」的假象），但**遮遮掩掩**比直接定价更伤信任
2. 文案末尾「**便于演示订阅与权限流程**」——这一句让用户觉得这是个**演示样品**，不是真产品（"演示订阅"暗示订阅本身是演示，不是真在卖）

**解决方案**：
- **立刻删掉「便于演示订阅与权限流程」这 11 个字**——它把整个产品自降级为演示样品
- 改成实操：每个员工卡片右上角加「免费 / 付费」标识，付费员工的卡片直接显示「**¥X/月 · 试用 7 天**」
- 推荐分层设计：**Navigator 永久免费 + 其他 5 位每月 3 次免费体验**——用 Navigator 做引流入口，其他 5 位做付费转化

### 🆕 发现 3（v3 已修正）：每位员工都有 skill 清单，但**执行过程对用户完全黑盒**

**实测**：
登录后访问 `/team/<agent>`（每位员工都有独立 URL），每位员工的详情页都有 Tab 结构：
**`总览 / Skills / 消息渠道 / 工作任务 / 文件`**（Sophia 还多了 `客服后台`）。

每位员工都有「**快速指令**」分组（这是产品里的 skill / 工具入口），实测清单：

| 员工 | skill 数 | 具名 skill 清单 |
|---|:---:|---|
| Alice（增长营销） | 5 | 整理推广成效、生成推文内容、生成广告文案、竞品对比分析、规划上线策略 |
| Sophia（客服） | 3 | 配置品牌与话术、管理知识库、查看对话分析 |
| William（品牌） | 4 | 检查品牌名+市场、生成 DNA 报告、做 GEO 体检、帮我建品牌 |
| Daniel（财务） | 5 | 分析财务文档、生成财务诊断、生成财务报告、决策影响分析、设置财务预警 |
| Ethan（法务） | 6 | 审查合同、翻译合同、对比多份合同、查询海外法规、监控法规变化、生成合同模板 |
| Navigator | — | 总览页展示「Navigator 诊断」（含 Twitter / LinkedIn / SEO 状态）+「重新诊断」按钮 |

**总共 23+ 个 skill 真实存在**——产品工具体系比 v2 想象的完整得多。

**真实判断**（v2 撤回，v3 重写）：

❌ ~~v2 说「AI 员工不会调工具」~~ — **撤回**。产品确实做了 skill 体系。

✅ 真正的问题是：**skill 的执行过程对用户完全黑盒**。具体表现：

1. **没有 tool-call 调用过程展示** — 用户点「分析财务文档」后，看不到「Daniel 正在调 PDF 解析工具 → 读取了 X.pdf → 提取出 6 维数据 → 调用财务模型」这条调用链。AI 黑盒地给出结果，用户不知道它做了什么。

2. **没有上传/输入控件可见** — 「分析财务文档」承诺解析 PDF/Excel/CSV，但页面上没有直接的「上传」按钮；「检查品牌名」有 `[名称] 去 [市场]` 的占位符，但是是自然语言填空，不是结构化表单。

3. **「动态」流全部「暂无动态」** — 6 位员工详情页都有「动态」区，理应记录历史 skill 调用。但全部空白——这表示要么用户没真正触发过 skill（停留在 chat），要么调用了但没记录。两种情况都说明**执行 trace 缺失**。

4. **没有数据源/集成披露** — 「查询海外法规」「检查域名可用性」「监控法规变化」需要外部数据源（LexisNexis / WHOIS / 政府公报 API），但页面上没有「已连接 / 未连接」的状态徽章，也没有 API key 配置入口。

5. **OKR 与 skill 解耦** — KR3.1（月度 P&L 报告 0→3）应该由「生成财务报告」skill 驱动，但两者之间**没有可见的关联 UI**。KR 进度更像手填数字，不是 skill 自动回写。

6. **快速指令更像「预设 prompt」而非「带 schema 的 tool」** — 命名都是动宾短语（"生成 / 汇总 / 分析"），不是 `post_tweet / fetch_metrics` 这种 API 风格；没有参数 schema、没有输入表单、没有版本号。判断上**应该是 skill 真有，但 UI 包装成了对话快捷键**。

**这意味着**：Octok 的工具能力是「**做了 80%，展示了 40%**」——后端可能真在调工具，前端把它当成对话蒙在鼓里。这比"没工具"还棘手，因为用户**不知道自己拥有什么**。

**解决方案**（修正版）：

- **短期（2 周内）**：给所有 skill 的执行加一条「**调用过程可视化**」：
  - 点击「分析财务文档」 → 弹出文件选择器（显式上传） → 进度条「📄 已上传 X.pdf → 🔍 解析中(读取 23 页) → 📊 提取 6 维数据 → ✅ 完成」
  - 在对话流里插入工具调用气泡（参考 ChatGPT/Claude 的 tool-call UI）
- **短期（1 个月内）**：「动态」流真做出来——每次 skill 调用都留 trace，用户能回看历史
- **中期（2-3 个月）**：对外暴露集成清单——dashboard 加「**已连接的数据源**」面板（哪些 API / 哪些数据库 / 哪些第三方授权），让用户清楚 AI 是基于真实数据还是 LLM 知识库猜测
- **中期（3-6 个月）**：把 OKR 进度与 skill 调用挂钩——「KR3.1 33%」旁边显示「由 Daniel 的 `生成财务报告` 自动更新于 X 时」
- **长期（6-12 个月）**：选 1-2 个 skill 做到真正的端到端执行（比如 Ethan 的「查询海外法规」真的接 LexisNexis 而不是 LLM 凭知识回答；Daniel 的「设置财务预警」真的能跑后台 cron）

**对官网首页的影响**：
- 「他们执行」这句话**现在算是部分兑现**（skill 是真的能跑），可以保留
- 但需要补一句**信任锚**：「**所有执行步骤都对你可见 → 工具调用 trace + 数据来源标注**」——把工具透明度做成卖点

### 🆕 发现 4：知识库结构是个**没解决的产品问题**（不是单纯的故障）

**实测**：
用户上传文件给 Daniel → 三轮搜不到 → Daniel 给了 4 种排查理由：
1. 文件刚上传，还在处理中 — 索引需要几分钟
2. 上传到了其他员工的知识库 — 需要确认是放在 Daniel 可访问的共享知识库里
3. 文件格式问题 — 某些格式可能未被正确索引
4. 文件夹下上传 vs 沙箱文件系统 — 平台同时存在 4 套存储概念

**真实判断**：
这是个**架构层面的结构问题**——Octok 自己没想清楚知识库怎么组织，**同时存在 4 套独立的存储概念**：

| 存储类型 | 谁能访问 | 什么时候用 | 用户能看到状态吗 |
|---|---|---|---|
| 对话附件 | 只有当前对话的那位员工 | 临时上下文 | ⚠️ 不明 |
| 单个员工的私有知识库 | 单个员工 | 这个员工的长期记忆 | ⚠️ 不明 |
| 共享知识库 | 所有 6 位员工 | 公司公共知识 | ⚠️ 不明 |
| 沙箱文件系统 | 员工代码执行环境 | 临时文件 | ❌ 用户完全看不见 |

用户被迫**学习这套存储模型**才能让 AI 员工找到文件——但用户的心智模型只有一个：「**我上传了，AI 就应该知道**」。当 AI 不知道时，信任就崩了。

**解决方案**：
- **把 4 套合并为 1 套**：所有上传都进「公司知识库」，6 位员工都能访问
- 上传界面加状态指示：`📤 上传中 → 🔍 索引中 → ✅ 已就绪（3 位员工已索引）`
- 每次对话里明示「📚 这次回答用到了：X.pdf / Y.md」说明知识来源
- 删掉「沙箱文件系统」这个用户可见的概念（保留作为底层实现细节，但不对外暴露）

### 🆕 发现 5：「5/6 员工运行中 / 12 个任务进行中」是演示文案，不是真功能

**实测**：
- 公开页演示截图显示「5/6 员工运行中 / 12 个任务进行中 / 状态健康」
- 登录后真实工作台**根本没有这个状态栏**——既不显示 5/6，也不显示任何任务计数
- 真实工作台就是 6 张员工卡片，没有运行状态

**真实判断**：
- 公开页的运行状态指标是**纯营销文案**，不是产品功能
- 这种「演示画面 vs 真实产品」的落差**只要被发现一次，整个营销页的可信度都会打折**
- 而且首页文案内部矛盾：同一处说「6 位 Agent 全部就绪」+「5/6 运行中」——内部话术不统一

**解决方案**：
- 要么真做状态栏（工作台顶部加「**3 个对话进行中 / 本周 12 个产出 / 6 位员工在线**」）
- 要么从首页演示截图里删掉「5/6 active」和「12 tasks in progress」这两个数字
- **推荐前者**——这种状态栏不难做，而且对用户留存有正面作用（"我的 AI 团队在帮我工作"的心智）

### 🆕 发现 6：登录后依然 0 个 定价 / 计费 入口

**实测**：
- 公开页：没有定价页（v1 已发现）
- 登录后：工作台顶部导航「控制台 / 工作日历 / AI 员工 / 企业资料 / 能力 / 新增对话」**没有「设置」「账单」「套餐」**
- 页脚：仍然只有「隐私政策 / 服务条款」，没有「定价」

**真实判断**：
这比 v1 的判断更糟糕——v1 推测「私域围墙是自残」，登录后实测发现**Octok 其实根本没建计费系统**：
- 用户即使想付费也没法主动给钱
- 「**部分员工需先雇佣后解锁**」承认了门禁存在，但门禁界面缺失
- 「雇佣 Navigator」按钮点了之后大概率是「敬请期待」或对话弹窗

这意味着 Octok 现在的**营收 = 0**。所有用户都是免费试用 / 邀请制。这对早期隐身阶段可以接受，但**离「产品-市场匹配」验证还有距离**——因为没人为它真金白银付过费。

**解决方案**：
最迟 30 天内做出最基础的计费系统：
1. 接入 Stripe（或 Lemon Squeezy / Paddle，新加坡公司用 Paddle 最方便）
2. 三档套餐页（免费 / ¥699/月 / ¥3500/月）
3. 工作台顶部加「升级」按钮
4. 免费档用 Navigator + Alice 即可，付费档解锁其他 4 位员工

**不要做的事**（避免过度工程）：
- ❌ 不要做按用量计费（按 token / 按任务计费）——太复杂，不适合早期
- ❌ 不要做企业大客销售——先用「产品自驱动增长」的路子跑出 100 个付费用户再说

### 🆕 发现 7（v3 已修正）：URL 路由**部分做了**——agent 详情页和对话有独立 URL，但主导航没有

**实测**（v3 重新验证）：
✅ **已有独立 URL 的页面**：
- 每位员工详情页：`/team/navigator` / `/team/alice` / `/team/sophia` / `/team/william` / `/team/daniel` / `/team/ethan`
- 对话会话：`/chat/<session-id>?agent=daniel`
- 进入工作台后地址变成 `/chat`

❌ **仍然没有独立 URL 的页面**：
- 「控制台」「工作日历」「企业资料」「能力」「新增对话」这 5 个主导航项点击后地址栏不变（仍是 `/chat`）
- 营销首页 + 工作台 dashboard 共用 `https://www.octok.com/` 这一个网址
- 营销首页本身不区分登录态，登录用户回到 `/` 还是先看营销，必须点「进入工作台」

**真实判断**（v2 撤回，v3 重写）：

❌ ~~v2 说「全部根本没网址」~~ — **过于绝对**。真实情况是 URL 路由做了一半：
- ✅ **做对了**：`/team/<agent>` 和 `/chat/<session-id>` 都是独立 URL，可分享、可收藏、可被搜索引擎收录
- ❌ **没做完**：主导航的 5 个 tab（控制台/工作日历/企业资料/能力/新增对话）没有 URL 同步
- ❌ **没解决**：营销首页 + 工作台共用 `/`——用户没法把工作台主页直接发给同事

**这意味着**：产品已经在做正确的事，但落地不彻底——是个**工作量小、价值大**的修复，不是大架构改动。

**解决方案**（修正版）：

- **优先级 P1（**1 周内**可做完）**：
  - 给主导航 5 个 tab 各自一个独立 URL：`/dashboard`、`/calendar`、`/profile`、`/capabilities`、`/new-chat`
  - 登录用户访问 `/` 时自动重定向到 `/dashboard`（不要再让他们看营销页）
  - 营销首页改成只在未登录态展示，登录态首页 = 工作台首页
- **优先级 P2（半年内）**：把工作日历的事件、企业资料的字段、能力页的内容都各加独立 URL（深链接），让产品**全栈可分享**——这是从「内部工具」升级为「协作平台」的必经一步

## 我**特别不建议**的事（基于登录后看到的）

- ❌ **不要急着加新员工**——现在 6 位里有 5 位被付费墙锁着，再加只是多几个空名字
- ❌ **不要做日历集成（Google / Outlook）**——内部工作日历自己都还是演示画面，先把内部的真做出来
- ❌ **不要做多组织 / 团队工作区**——单用户体验都还没立住，团队功能是分散精力
- ❌ **不要追求 Product Hunt 上线发布**——现在发布，5/6 员工锁着 + 知识库找不到文件 = 一波负面评价

## 一句话总结（给 CEO 看）

> Octok 是一个**能力做了 80%、展示了 40%** 的产品。当前最大的风险不是「功能不够」，
> 是「**做了的工具用户看不见**」——23+ 个 skill 全在但执行过程黑盒、动态流空白、调用 trace 缺失，
> 用户和潜在客户都不知道产品已经做到哪一步了。最快的修复路径不是加功能，是**给现有 skill 加执行可视化**——
